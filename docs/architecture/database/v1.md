# OUTFLO — DATABASE CONTRACT (v1)

This document defines the database contracts and invariants for Outflō.

Purpose:
- Prevent schema drift
- Define canonical entities and relationships
- Lock invariants for ingest, identity, and ledger derivations

Database naming follows `snake_case`. :contentReference[oaicite:1]{index=1}

---

## 1. Scope

In scope:
- Canonical tables and relationships
- Row ownership and access expectations
- Ingest idempotency requirements
- Ledger derivation rules (what is stored vs computed)

Out of scope:
- UI structure
- Non-production experiments
- Exact index tuning (unless required by invariants)

---

## 2. Global Rules (Locked)

### 2.1 Naming
- Tables, columns, constraints: `snake_case`. :contentReference[oaicite:2]{index=2}
- Primary keys SHOULD be `id` (UUID) unless the table is explicitly keyed differently.

### 2.2 Timestamps
- Tables SHOULD include:
  - `created_at` (`timestamptz`, default now())
  - `updated_at` (`timestamptz`, maintained by trigger or application)
- Event tables MAY omit `updated_at` if immutable.

### 2.3 Ownership
- Every user-owned row MUST be attributable to a user via `user_id` (UUID referencing auth user).
- User-owned tables MUST be protected by RLS such that:
  - A user can only access rows where `user_id = auth.uid()` (or equivalent policy).

### 2.4 Immutability vs Mutation
- Ingest/event rows SHOULD be append-only (immutable).
- Derived/summary rows MAY be recomputed, but must preserve deterministic meaning.

## 3. Canonical Entities

> Note: This section defines **contractual intent**. If a table exists with a different name today,
> treat this as the target mapping and align by migration when appropriate.

### 3.1 Users (Auth)
- Canonical identity lives in Supabase Auth.
- Application tables reference users via `user_id` (UUID).

---

### 3.2 `user_epoch` (Identity-Bound Time Anchor)
Purpose:
- Store the user’s canonical epoch anchor (identity-bound, not device-bound).

Contract:
- There MUST be exactly **one active** epoch row per user.
- Epoch MUST be created once at activation (or first-use) and persist across devices.
- Application time calculations MUST derive from the stored epoch (no local fallback).

Minimum fields (recommended):
- `id` (uuid, pk)
- `user_id` (uuid, unique, fk → auth user)
- `epoch_ms` (bigint) OR `epoch_ts` (timestamptz) — choose one and stay consistent
- `created_at` (timestamptz)

Invariants:
- `user_epoch.user_id` is UNIQUE.
- Epoch value MUST NOT drift unless explicitly reset via admin tooling.

---

### 3.3 `receipts` (Canonical Receipt Record)
Purpose:
- Store normalized receipt records that can be displayed and referenced by UUID routes.

Contract:
- Each receipt row MUST belong to exactly one user (`user_id`).
- Each receipt MUST have a stable UUID `id`.
- Receipts MAY be created via:
  - email ingest
  - manual entry
  - admin import/restore
- Receipts SHOULD preserve raw source metadata sufficient to re-parse or audit.

Minimum fields (recommended):
- `id` (uuid, pk)
- `user_id` (uuid, fk)
- `amount_cents` (int) OR `amount` (numeric) — choose one standard
- `currency` (text, default "USD")
- `merchant_name` (text, nullable)
- `merchant_slug` (text, nullable)
- `purchased_at` (timestamptz, nullable)
- `source` (text) — e.g. "email", "manual", "import"
- `raw_payload` (jsonb, nullable) — original parse payload / audit blob
- `created_at` (timestamptz)

Invariants:
- `receipts.user_id` MUST be present.
- Receipt `id` MUST be used for routing (`/app/money/receipts/[id]`).
- Amount MUST be non-negative (refunds/credits, if supported later, should be explicit).

---

### 3.4 `ingest_events` (Idempotent Ingest Log)
Purpose:
- Record inbound ingest attempts (e.g., Resend webhook) for traceability and idempotency.

Contract:
- Ingest MUST be idempotent.
- There MUST be a stable external identifier (e.g. `email_id`, `message_id`, provider event id).
- Replays must not create duplicate receipts.

Minimum fields (recommended):
- `id` (uuid, pk)
- `provider` (text) — e.g. "resend"
- `provider_event_id` (text, unique) OR (`message_id` unique) — pick the real key
- `user_id` (uuid, nullable if not yet resolved)
- `status` (text) — e.g. "received" | "processed" | "skipped" | "failed"
- `error` (text, nullable)
- `payload` (jsonb) — original webhook payload
- `created_at` (timestamptz)

Invariants:
- Unique constraint on the chosen idempotency key.
- Processing a duplicate event MUST be a no-op.

---

### 3.5 `outflows` (Ledger Entries)
Purpose:
- Represent the canonical ledger entries used for rollups (day, place, 365).

Decision contract (pick one and keep it stable):
- Option A (recommended): receipts are canonical and outflows are derived (view/materialization).
- Option B: outflows are canonical and receipts attach as metadata.

Current default contract:
- Outflō treats **spend events as ledger entries** and renders views from them.
- If receipts are the primary ingestion source, the system MAY store only receipts
  and derive outflows from them deterministically.

Minimum fields (recommended if stored):
- `id` (uuid, pk)
- `user_id` (uuid, fk)
- `amount_cents` (int) OR `amount` (numeric)
- `occurred_at` (timestamptz)
- `merchant_slug` (text, nullable)
- `receipt_id` (uuid, nullable, fk → receipts)
- `created_at` (timestamptz)

Invariants:
- Ledger rollups MUST be reproducible from the canonical source (receipts or outflows).
- Any denormalized rollup MUST be derivable and correctable.

## 4. Derivation Surfaces (Read Models)

These are conceptually part of the database contract even if implemented in application code.

### 4.1 Day rollup
- Keyed by `day_key` (contract: stable string key, e.g. `YYYY-MM-DD` or epoch-day index)
- Derived from canonical ledger entries.

### 4.2 Place rollup
- Keyed by `merchant_slug`
- Derived from canonical ledger entries.

### 4.3 365 window
- Rolling window aggregate derived from canonical ledger entries.

---

## 5. Security Model

### 5.1 RLS (Required)
- User-owned tables MUST enforce RLS.
- Admin operations MUST be isolated to privileged routes and service role usage.

### 5.2 Service Role Use (Restricted)
- Service role access is permitted only in server-only contexts (route handlers),
  never in client code.

---

## 6. Operational Guarantees

### 6.1 Idempotency
- Ingest pipeline MUST be safe under:
  - retries
  - duplicate webhook deliveries
  - delayed delivery

### 6.2 Auditability
- Raw payloads SHOULD be retained (jsonb) for ingest-based creation.
- Admin resets MUST be logged as events (even if only internally).

---

## 7. Definition of Done (Database Contract)

This contract is considered “implemented” when:
- All canonical tables exist (or are mapped cleanly)
- RLS protects user-owned data
- Ingest is idempotent by constraint
- Receipts resolve by UUID consistently
- Epoch is identity-bound and stable across devices
